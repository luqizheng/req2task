# Bug修复规则

## 问题记录

发现Bug时，必须记录以下信息：

- **复现步骤**：清晰描述触发Bug的操作序列
- **预期行为**：用户期望的正常结果
- **实际行为**：当前系统的错误表现
- **环境信息**：操作系统、浏览器版本、Node版本、数据库状态等
- **错误日志**：控制台错误、服务器日志、数据库查询日志
- **严重程度**：P0（系统崩溃）/ P1（核心功能失效）/ P2（功能异常）/ P3（体验问题）

## 根因分析

修复前必须完成根因分析：

1. **定位代码**：通过错误堆栈、日志、调试断点确定问题代码位置
2. **分析链路**：追踪数据流和调用链，找出根本原因
3. **复现验证**：在本地环境完整复现问题
4. **影响评估**：确认修复范围，评估对其他功能的影响

禁止：
- 未定位根因就盲目修改代码
- 只修复表面症状，不解决根本问题
- 修复时引入新的副作用

## 修复流程

### 1. 创建修复分支

```bash
git checkout -b fix/{issue-id}-{简短描述}
```

### 2. 编写或更新测试

- 为Bug编写可复现的单元测试或集成测试
- 测试应验证修复后的正确行为
- 确保测试覆盖边界情况

### 3. 实施修复

- 修改代码解决根因
- 遵循项目代码风格和规范
- 保持代码简洁，避免过度工程
- 使用结构化日志记录关键操作

### 4. 验证修复

- 运行相关单元测试
- 运行集成测试验证功能
- 手动复现问题，确认修复有效

## 代码规范

- 修复代码必须通过 ESLint 检查
- 修复代码必须通过类型检查（tsc --noEmit）
- 禁止使用 any 类型掩盖问题
- 禁止使用 @ts-ignore 跳过类型检查
- 修复后运行 `pnpm lint` 和 `pnpm test`

## 提交规范

提交信息格式：

```
fix({scope}): {简短描述}

- {修复内容1}
- {修复内容2}
```

示例：

```
fix(user-module): 修复用户登录后token过期未刷新问题

- 添加token自动刷新逻辑
- 新增token过期前5分钟刷新机制
```

## 回滚预案

每个修复必须包含回滚方案：

- 如果是数据库修改，准备回滚SQL脚本
- 如果是配置修改，记录原始配置值
- 如果是多文件修改，确保可以单次提交回滚

## 验证清单

修复完成后确认：

- [ ] 本地环境问题已复现并修复
- [ ] 相关单元测试通过
- [ ] 相关集成测试通过
- [ ] ESLint检查通过
- [ ] TypeScript类型检查通过
- [ ] 无控制台错误或警告
- [ ] 核心业务流程验证通过
- [ ] 提交信息符合规范
