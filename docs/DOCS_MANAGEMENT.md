# 文档管理规则

## 一、文档分类体系

### 1.1 七层架构

| 层级 | 目录 | 职责 | 变更频率 | 负责人 |
|------|------|------|----------|--------|
| **架构层** | `architecture/` | 系统架构、技术选型、模块关系 | 极低 | 架构师 |
| **设计层** | `design/` | 数据库设计、模块设计、AI设计 | 低 | 技术负责人 |
| **需求层** | `prd/` | 业务模型、功能流程、需求规范 | 中 | 产品经理 |
| **规范层** | `conventions/` | 命名规范、编码规范、错误处理 | 低 | 全员 |
| **计划层** | `plans/` | Sprint 计划、里程碑、资源分配 | 高 | 项目经理 |
| **手册层** | `manuals/` | 用户手册、管理员指南 | 中 | 文档工程师 |
| **参考层** | `reference/` | API 规范、错误码、自动生成 | 高 | 自动生成 |

### 1.2 各层文档职责

#### architecture/ - 架构层
```
architecture/
├── overview.md           # 系统整体架构图
├── system-overview.md    # 产品定位与目标用户
└── data-flow.md          # 数据流转图
```
**规则**：
- 架构层文档需配架构图（PNG/SVG）
- 重大架构变更需创建新版本文件，保留旧版
- 每半年审视一次架构文档有效性

#### design/ - 设计层
```
design/
├── _meta/                # 版本元数据
├── ai/                   # AI 能力设计
├── data/                 # 数据库设计
├── modules/              # 模块边界设计
└── project/              # 项目相关设计
```
**规则**：
- 每个子目录需有 `readme.md` 索引
- 设计变更需记录变更日志
- 数据库设计需包含 ER 图和迁移脚本

#### prd/ - 需求层
```
prd/
├── index.md              # PRD 索引
├── system-overview.md    # 系统概述
├── core-models.md        # 核心业务模型
├── functional-flows.md   # 功能流程
└── *.md                  # 其他需求文档
```
**规则**：
- 每个功能模块独立文件
- 需求变更使用 `diff` 标记
- 验收标准与测试用例关联

#### conventions/ - 规范层
```
conventions/
├── naming.md              # 命名规范
├── error-handling.md      # 错误处理规范
└── logging.md            # 日志规范
```
**规则**：
- 规范文档需包含代码示例
- 示例需标注语言类型
- 违反规范需有明确后果说明

#### plans/ - 计划层
```
plans/
├── index.md              # Sprint 索引总览
├── sprint-XX-*.md        # 单个 Sprint 计划
└── backlog.md            # 产品待办列表
```
**规则**：
- Sprint 计划命名：`sprint-XX-{主题}.md`
- 每个 Sprint 完成后更新状态
- 使用状态符号：✅ 完成 | 🚧 进行中 | ⏳ 待开始

#### manuals/ - 手册层
```
manuals/
├── index.md              # 手册索引
├── user-management.md    # 用户管理指南
└── admin-permissions.md  # 管理员指南
```
**规则**：
- 按角色分类（最终用户、管理员、开发者）
- 需包含操作步骤截图或视频链接
- 每季度更新一次

#### reference/ - 参考层
```
reference/
├── api-spec.md           # API 规范（自动生成）
└── error-codes.md        # 错误码定义
```
**规则**：
- 优先从代码自动生成
- 手动维护需标注 "MANUAL" 标记
- API 变更需同步更新版本号

## 二、文件命名规范

### 2.1 命名原则

| 场景 | 规则 | 示例 |
|------|------|------|
| 普通文档 | kebab-case | `requirement-priority.md` |
| Sprint 计划 | `sprint-{序号}-{类型}.md` | `sprint-01-frontend-projects-modules.md` |
| 模块文档 | `{模块}-{子主题}.md` | `database-design.md` |
| 索引文件 | `index.md` 或 `readme.md` | `design/readme.md` |
| 临时文档 | `_temp-{描述}.md` | `_temp-draft-api.md` |

### 2.2 禁止的命名

- 中文文件名（临时草稿除外）
- 空格
- 特殊字符（`#`, `*`, `?`）
- 超过 50 字符的文件名

## 三、元数据规范

### 3.1 Frontmatter 必填项

```yaml
---
title: 文档标题
description: 一句话描述文档用途
last_updated: YYYY-MM-DD
status: draft | review | active | deprecated
owner: 负责人/团队
---
```

### 3.2 状态定义

| 状态 | 含义 | 使用场景 |
|------|------|----------|
| `draft` | 草稿 | 正在编写中 |
| `review` | 评审中 | 等待 Review |
| `active` | 生效中 | 正式使用 |
| `deprecated` | 已废弃 | 被新文档替代 |

### 3.3 可选元数据

```yaml
---
version: 1.0.0           # 文档版本
requires_review: true    # 是否需要定期审视
review_interval: 6month  # 审视周期
related_docs:            # 关联文档
  - prd/core-models.md
  - design/database-design.md
tags:                    # 标签
  - AI
  - RAG
  - 需求管理
---
```

## 四、维护规则

### 4.1 责任人制度

| 文档类型 | 第一负责人 | 第二负责人 | 备份负责人 |
|----------|------------|------------|------------|
| architecture/* | 架构师 | 技术负责人 | 项目经理 |
| design/* | 技术负责人 | 核心开发 | 产品经理 |
| prd/* | 产品经理 | 技术负责人 | 项目经理 |
| conventions/* | 技术负责人 | 全员维护 | - |

### 4.2 更新触发条件

| 触发条件 | 需更新的文档 |
|----------|--------------|
| 代码合并到 main | reference/* |

### 4.3 更新频率

| 文档类型 | 最长更新周期 | 强制审视周期 |
|----------|--------------|--------------|
| architecture | 12 个月 | 6 个月 |
| design | 6 个月 | 3 个月 |
| prd | 3 个月 | 1 个月 |
| conventions | 6 个月 | 6 个月 |
| plans | 每个 Sprint | 每个 Sprint |
| manuals | 3 个月 | 1 个月 |
| reference | 自动更新 | - |

## 五、版本控制

### 5.1 版本号规范

使用语义化版本：`主版本.次版本.修订号`

| 变更类型 | 版本号变化 | 示例 |
|----------|------------|------|
| 重大重构 | 主版本 +1 | v1.0.0 → v2.0.0 |
| 新增功能 | 次版本 +1 | v1.0.0 → v1.1.0 |
| 修正错误 | 修订号 +1 | v1.0.0 → v1.0.1 |

### 5.2 变更日志要求

每个文档底部需包含变更日志：

```markdown
## 变更日志

| 版本 | 日期 | 变更人 | 变更内容 |
|------|------|--------|----------|
| v1.1.0 | 2026-04-17 | 张三 | 新增 AI 冲突检测章节 |
| v1.0.0 | 2026-01-01 | 李四 | 初始版本 |
```

### 5.3 版本对比

- 保留最近 3 个大版本的文档
- 旧版本移至 `archive/` 目录
- 归档文件命名：`{原文件名}-v{版本}.md`

## 六、质量标准

### 6.1 文档质量检查清单

| 检查项 | 要求 | 权重 |
|--------|------|------|
| Frontmatter 完整 | 必填项齐全 | 必须 |
| 标题层级清晰 | 最多 3 级标题 | 必须 |
| 内容结构化 | 使用表格、列表、代码块 | 80% |
| 链接有效性 | 所有内部链接可访问 | 必须 |
| 示例可用 | 代码示例可复制执行 | 60% |
| 配图标注 | 图片有 alt 文字 | 50% |

### 6.2 禁止出现的内容

- 过期信息（超过 30 天未更新且已变更）
- 死链接
- 模板占位符（TODO、FIXME 等需标注）
- 敏感信息（密码、密钥、内网地址）
- 中文与英文混排（专有名词除外）

### 6.3 可读性要求

| 类型 | 最大行数 | 建议段落长度 |
|------|----------|--------------|
| 索引文件 | 150 行 | 每段 ≤ 5 行 |
| 详细文档 | 500 行 | 每段 ≤ 5 行 |
| 单个章节 | 100 行 | - |

## 七、评审流程

### 7.1 文档评审触发条件

| 场景 | 评审范围 | 评审人 |
|------|----------|--------|
| 新增文档 | 全部 | 文档所有者 + 1 名相关方 |
| 重大变更 | 变更部分 | 文档所有者 + 相关技术负责人 |
| 定期审视 | 全部 | 文档所有者 |
| 请求 Review | 全部 | 申请人指定的 Reviewer |

### 7.2 评审检查点

1. **准确性**：内容与代码实现一致
2. **完整性**：覆盖所有必要场景
3. **可读性**：结构清晰、语言简洁
4. **可维护性**：便于后续更新
5. **链接完整性**：无死链

### 7.3 评审状态标记

在文档顶部使用评审标记：

```markdown
<!--
评审状态: 已评审
最后评审: 2026-04-17
评审人: 技术负责人
-->
```

## 九、工具与自动化

### 9.1 自动化任务

| 任务 | 工具 | 触发条件 |
|------|------|----------|
| 链接检查 | markdown-link-check | PR 时 |
| 文档索引更新 | 自定义脚本 | 合并到 main |
| API 文档生成 | swagger/OpenAPI | 代码提交时 |
| 文档搜索索引 | Algolia/Lunr | 文档更新时 |

### 9.2 文档验证脚本

项目根目录提供文档验证脚本：

```bash
pnpm docs:check     # 检查文档完整性
pnpm docs:links      # 检查链接有效性
pnpm docs:lint       # 检查文档格式
pnpm docs:index      # 更新索引文件
```

## 十、归档与清理

### 10.1 归档条件

满足以下任一条件移入归档：

- 版本超过 3 个大版本
- 对应功能已下线
- 内容超过 12 个月未更新且无计划更新
- 被新文档完全替代

### 10.2 归档目录结构

```
docs/
├── archive/                 # 归档目录
│   ├── 2026-Q1/            # 按季度归档
│   │   ├── prd-v1.0.md
│   │   └── design-v1.0.md
│   └── 2025/
│       └── ...
```

### 10.3 清理规则

| 文档类型 | 最大保留版本 | 超期处理 |
|----------|--------------|----------|
| architecture | 2 个 | 归档 |
| design | 3 个 | 归档 |
| prd | 3 个 | 归档 |
| plans | 当前 + 2 个 | 删除旧版 |
| reference | 自动生成 | 覆盖旧版 |

## 十一、快速参考

### 11.1 创建新文档模板

```markdown
---
title: 文档标题
description: 一句话描述
last_updated: YYYY-MM-DD
status: draft
owner: 负责人
---

# 文档标题

## 概述

## 详细内容

## 相关文档

## 变更日志

| 版本 | 日期 | 变更人 | 变更内容 |
|------|------|--------|----------|
```

### 11.2 常用命令

| 命令 | 说明 |
|------|------|
| `pnpm docs:check` | 检查所有文档完整性 |
| `pnpm docs:links` | 验证内部链接 |
| `pnpm docs:lint` | 检查格式规范 |
| `pnpm docs:index` | 生成/更新索引 |

### 11.3 状态速查

| 符号 | 状态 | 用途 |
|------|------|------|
| ✅ | 已完成 | 功能/任务完成 |
| 🚧 | 进行中 | 正在开发 |
| ⏳ | 待开始 | 尚未开始 |
| ⚠️ | 有变更 | 存在待处理变更 |
| ❌ | 已取消 | 已取消 |

## 十二、违规处理

| 违规类型 | 处理方式 |
|----------|----------|
| 未按规范命名 | PR 被拒绝，要求重命名 |
| 缺少 Frontmatter | PR 被拒绝，要求补充 |
| 链接失效 | PR 被拒绝，要求修复 |
| 超过行数限制 | 要求拆分 |
| 未更新索引 | 合并前自动更新 |
