# 文档编写规则

## 1. 文档分类与用途

### 1.1 产品需求文档（PRD）

- **位置**：`docs/prd/index.md`
- **用途**：描述产品功能、用户需求、业务流程、数据模型
- **读者**：产品经理、项目经理、开发团队、测试团队
- **内容要求**：
  - 使用业务语言，避免技术术语
  - 描述"做什么"，不描述"怎么做"
  - 使用表格、流程图、层级结构等可视化方式
  - 包含用户角色、使用场景、核心价值

### 1.2 设计文档

- **位置**：`docs/设计/` 目录
- **用途**：描述技术实现方案、系统架构、API设计、数据库设计
- **读者**：开发工程师、架构师、技术负责人
- **内容要求**：
  - 包含技术实现细节
  - 使用UML图、ER图、时序图等
  - 包含API接口定义、数据库表结构
  - 描述"怎么做"，包含技术选型、实现方案

### 1.3 其他文档

- **API文档**：`docs/api/` - OpenAPI/Swagger格式
- **部署文档**：`docs/deployment/` - 部署指南、环境配置
- **用户手册**：`docs/user-guide/` - 最终用户使用指南

## 2. 编写原则

### 2.1 分离关注点

- **PRD中不写技术细节**：不包含SQL、API参数、代码实现
- **设计文档不写业务需求**：专注于技术实现方案
- **单一职责**：每个文档有明确的目的和读者

### 2.2 结构化组织

- 使用清晰的层级结构（H1-H4）
- 使用表格整理结构化数据
- 使用代码块展示示例、配置
- 使用列表描述步骤、要点

### 2.3 可维护性

- 保持文档与代码同步更新
- 使用版本控制跟踪文档变更
- 定期审查和更新文档

```
### 1.2 目标用户

### 1.4 使用场景

## 2. 核心模型

### 2.2 实体关系
### 2.3 实体字段定义

## 3. 核心功能流程
### 3.1 [功能名称]流程
### 3.2 [功能名称]流程

## 4. API接口设计（概要）
### 4.1 [模块名称]管理
### 4.2 [模块名称]管理
```


## 4. 命名规范

### 4.1 文件命名

- 中文文档：使用中文名称，如`产品需求文档.md`
- 技术文档：使用英文名称，如`api-design.md`
- 统一使用小写字母、连字符分隔

### 4.2 目录结构

```
docs/
├── 产品需求文档.md      # 主PRD
├── agents.md           # 文档编写规则
├── design/             # 设计文档
│   ├── database-design.md
│   ├── api-design.md
│   └── architecture.md
├── api/                # API文档
│   ├── openapi.yaml
│   └── api-reference.md
├── deployment/         # 部署文档
│   ├── setup-guide.md
│   └── configuration.md
└── user-guide/         # 用户手册
    ├── quick-start.md
    └── advanced-features.md
```

## 5. 版本控制

### 5.1 变更记录

- 每个文档末尾添加变更记录
- 记录变更日期、变更人、变更内容

### 5.2 示例

```
## 变更记录

| 日期 | 版本 | 变更人 | 变更内容 |
|------|------|--------|----------|
| 2024-01-15 | v1.0 | 张三 | 初始版本 |
| 2024-02-01 | v1.1 | 李四 | 添加用户故事模型 |
```

## 6. 质量检查清单

### 6.1 内容检查

- [ ] 是否包含所有必要章节
- [ ] 是否使用正确的模板
- [ ] 是否分离了业务需求和技术实现
- [ ] 是否包含示例和图表

### 6.2 格式检查

- [ ] 标题层级是否正确
- [ ] 表格格式是否规范
- [ ] 代码块是否有语言标记
- [ ] 链接是否有效

### 6.3 一致性检查

- [ ] 术语使用是否一致
- [ ] 与代码实现是否一致
- [ ] 与其他文档是否一致

## 7. 工具支持

### 7.1 文档生成

- 使用 `typedoc` 生成TypeScript API文档
- 使用 `swagger-ui` 展示OpenAPI文档
- 使用 `mermaid` 生成流程图、时序图

### 7.2 质量检查

- 使用 `markdownlint` 检查Markdown格式
- 使用 `vale` 检查写作风格
- 使用 `pre-commit` 自动检查
